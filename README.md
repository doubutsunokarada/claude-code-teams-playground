# 多エージェント開発ガイド — TODO管理Webアプリ

7つの専門エージェントが連携してTODO管理Webアプリを構築するプレイグラウンド。開発記録はMarkdownファイルとして `blog/` ディレクトリに蓄積し、プロジェクト完了時に総評記事を1本まとめる。各エージェントの役割・開発フロー・PR運用ルールを定義する。

---

## エージェント一覧と担当タスク

このプロジェクトでは、`.claude/agents/` に定義された7つの専門エージェントが連携して開発を進めます。

### 1. Senior Project Manager (📝)

**職能**: 仕様書をもとに、開発タスクを具体的・実行可能な単位に分解するプロジェクト管理の専門家。スコープの肥大化を防ぎ、現実的な計画を立てる。

**担当タスク**:
- プロジェクト全体の要件整理と仕様策定
- GitHub Issuesにタスクを起票する（1タスク30〜60分で実装可能な粒度）
- 各Issueに受け入れ基準・ラベル・担当者を設定
- 技術スタック要件の整理
- 開発スケジュールと優先順位の管理

> **⚠️ 必須ルール**:
> - タスク管理はGitHub Issuesで行うこと。`gh issue create` でCLI経由で起票する。
> - **Backend Architect と Frontend Developer は必ず同時並行で起動すること。** Backendの完了を待ってからFrontendを開始するシリアルな進め方は禁止。両エージェントを同一メッセージ内の複数 tool use ブロックで同時起動し、並列開発を指示する。

---

### 2. Software Architect (🏛️)

**職能**: ドメイン駆動設計・アーキテクチャパターン・トレードオフ分析に精通したソフトウェア設計の専門家。チームが長期的にメンテナンスできるシステムを設計する。

**担当タスク**:
- TODO管理アプリの全体アーキテクチャ設計
- ドメインモデリング（TODO、カテゴリ、ユーザーなど）
- アーキテクチャパターンの選定（モノリス vs マイクロサービスなど）
- ADR（Architecture Decision Record）の作成
- データフローとコンポーネント間の依存関係の設計
- ローカル開発環境の構成設計（`compose.yaml` でDB・キャッシュ等のミドルウェアを定義）

> **⚠️ 必須ルール**:
> - 設計内容はGitHub Wikiにまとめること。Wikiへの記載完了後、Technical Writerがドキュメントを整備する。
> - **APIエンドポイント定義とドメインモデルを最優先で完成させ、`docs/architecture/domain-model.md` に出力すること。** このファイルが揃えば Backend・Frontend・Technical Writer が全員並行して動ける。詳細設計は後から追記してよいが、インターフェース定義だけは先行して確定させる。

---

### 3. Backend Architect (🏗️)

**職能**: スケーラブルなシステム設計・データベース設計・API開発・クラウドインフラの専門家。堅牢でセキュアなサーバーサイドアプリケーションを構築する。

**担当タスク**:
- データベーススキーマ設計（TODO、ユーザー、カテゴリテーブルなど）
- REST API / GraphQL APIの設計と実装
- 認証・認可システムの構築
- セキュリティ対策（レートリミット、入力バリデーション、暗号化）
- キャッシュ戦略とパフォーマンス最適化
- デプロイ環境とインフラ構成の設計
- `compose.yaml` の実装（DB・キャッシュ・その他ミドルウェアをコンテナで起動）
- APIのユニットテスト作成（ハンドラー・サービス層・バリデーションロジック）

> **⚠️ 必須ルール**:
> - Technical Writerのドキュメント整備完了後に実装を開始すること。
> - **ミドルウェア（DB・Redisなど）はすべて `compose.yaml` で管理すること。** ローカルに直接インストールしない。`docker compose up -d` 一発で開発環境が立ち上がる状態にする。
> - **ユニットテストを実装と同時に書くこと。** テストなしのPRはレビュー対象外。カバレッジの目安はビジネスロジック層80%以上。
> - 作業完了後は必ずPull Requestを作成する（直接mainブランチへのpushは禁止）。Approve後は自身で `gh pr merge` を実行してマージする。
> - **`git worktree` を使って独立したworktreeで作業すること。** mainブランチの作業ディレクトリと物理的に分離した環境で作業し、誤ったブランチへのpushを防ぐ。Agent tool の `isolation: "worktree"` オプションを使うか、`git worktree add` で手動セットアップする。
> - **PRを提出したら、レビューの完了を待たずに次のタスクへ先行着手すること。** Code Reviewerによるレビューは非同期で行われる。レビュー待ちで手を止めず、GitHub IssuesのBacklogから次のタスクを取得して作業を継続する。レビュー指摘が届いたらそちらを優先して対応する。

---

### 4. Frontend Developer (🖥️)

**職能**: React/Vue/Angularなどのモダンフレームワークを駆使し、レスポンシブでアクセシブル、高パフォーマンスなWebアプリケーションを構築するフロントエンド開発の専門家。

**担当タスク**:
- TODO管理画面のUI実装（一覧・作成・編集・削除・完了切り替え）
- レスポンシブデザインとモバイル対応
- Core Web Vitalsの最適化
- アクセシビリティ対応（WCAG 2.1 AA準拠）
- コンポーネントライブラリとデザインシステムの構築
- コンポーネント・カスタムフックのユニットテスト作成
- Playwrightによるe2eテスト作成（主要ユーザーフローをカバー）

> **⚠️ 必須ルール**:
> - Technical Writerのドキュメント整備完了後に実装を開始すること。
> - **ユニットテストを実装と同時に書くこと。** コンポーネント・カスタムフック・ユーティリティ関数を対象とし、テストなしのPRはレビュー対象外。
> - **Playwrightでe2eテストを書くこと。** `e2e/` ディレクトリにテストを配置し、TODO作成・編集・削除・完了切り替えなどの主要フローをカバーする。e2eテストはバックエンドが動いている状態（`docker compose up` 済み）で実行する。
> - 作業完了後は必ずPull Requestを作成する（直接mainブランチへのpushは禁止）。Approve後は自身で `gh pr merge` を実行してマージする。
> - **`git worktree` を使って独立したworktreeで作業すること。** Agent tool の `isolation: "worktree"` オプションを使うか、`git worktree add` で手動セットアップする。
> - **PRを提出したら、レビューの完了を待たずに次のタスクへ先行着手すること。** Code Reviewerによるレビューは非同期で行われる。レビュー待ちで手を止めず、GitHub IssuesのBacklogから次のタスクを取得して作業を継続する。レビュー指摘が届いたらそちらを優先して対応する。
> - **MSW（Mock Service Worker）を使い、バックエンドの完成を待たずに開発を進めること。** `src/mocks/` ディレクトリにAPIハンドラーを定義し、開発環境でのみ有効化する（`NEXT_PUBLIC_API_MOCKING=enabled`）。モックデータは `docs/architecture/domain-model.md` のAPIスキーマを参照して定義する。

### 5. Code Reviewer (👁️)

**職能**: 正確性・セキュリティ・保守性・パフォーマンスに焦点を当てた建設的なコードレビューを提供する専門家。指摘するだけでなく、開発者のスキル向上にも貢献する。

**担当タスク**:
- 全実装コードのレビュー（フロントエンド・バックエンド両方）
- セキュリティ脆弱性の検出（SQLインジェクション、XSS、認証バイパスなど）
- パフォーマンス問題の指摘（N+1クエリ、不要な再レンダリングなど）
- テストカバレッジの確認と改善提案
- コード品質の維持とベストプラクティスの推進

> **⚠️ 必須ルール**:
> - 他エージェントが作成したPull Requestを必ずレビューすること。レビュー未実施のPRはマージしない。
> - **PRが来たら速やかにレビューし、開発者をブロックしないこと。** Backend・Frontend の両PRは独立しているため、どちらかのレビュー待ちでもう一方を止めない。並行してレビューを進める。
> - 軽微な指摘で全体をブロックしない。致命的な問題（セキュリティ・バグ）のみ `request changes`、改善提案はコメントに留めてApproveする判断も取ること。

---

### 6. Technical Writer (📚)

**職能**: 開発者向けドキュメント・APIリファレンス・チュートリアルの作成に特化した技術ライティングの専門家。複雑な技術をわかりやすく正確に伝える。

**担当タスク**:
- Software Architectの設計完了後、GitHub Wikiの設計内容をもとに開発者向けドキュメントを整備
- APIドキュメントの作成（エンドポイント一覧、リクエスト/レスポンス例）
- セットアップガイド・開発環境構築手順の整備
- ユーザー向けヘルプドキュメントの作成
- コントリビューションガイドの作成

> **⚠️ 必須ルール**:
> - Software Architectの設計完了後にドキュメントを整備すること。
> - **全設計の完成を待たずに、`docs/architecture/domain-model.md`（APIスキーマ・ドメインモデル）が揃った時点で先行してAPIドキュメントを整備し、Backend・Frontend が動き出せる状態を作ること。** 詳細ドキュメントは後から追記してよい。ドキュメント整備をボトルネックにしない。

---

### 7. Book Co-Author (📘)

**職能**: このプロジェクトの**観測者**として、開発の経過・意思決定・学びをMarkdownで記録する記録者。プロダクトの機能紹介ではなく、「どう作ったか・何を学んだか・何に詰まったか」という開発プロセスそのものをコンテンツ化する。

**担当タスク**:
- 各エージェントの作業経過・PRのやり取り・設計判断を観察し、`blog/` ディレクトリにMarkdownで記録していく
- 技術的な意思決定（ADR・設計変更など）の背景とトレードオフをストーリーとして記事化
- 詰まった箇所・失敗・改善サイクルをあえて記録し、リアルな学習記録として残す
- **プロジェクト完了後に総評記事を1本書く**（全体を振り返り、学びと気づきをまとめた締めくくりの記事）

> **⚠️ 必須ルール**:
> - 記録はすべてMarkdown形式（`.md`）で `blog/` ディレクトリに保存すること。HTML直書きは不可。Webサイトとして公開する必要はない。
> - **プロダクトの機能紹介を書かないこと。** 開発プロセス・学び・判断の記録に徹する。
> - **他のエージェントの完了を待たずに、プロジェクト開始と同時に観察・記録を始めること。** PR・Wiki・Issueの動きを随時追いかけ、ネタが溜まった段階でMarkdownにまとめる。最終的な総評記事は開発完了後に1本書く。

## 開発フロー

各フェーズ内のエージェントは可能な限り並行して動かす。次のフェーズを待つ間も他の作業を止めない。

```
【フェーズ1】同時並行
  PM            → GitHub Issues にタスクを起票
  Book Co-Author → 観察・記録を開始（プロジェクト全期間通じて継続）

【フェーズ2】
  Software Architect → 全体設計・ドメインモデル定義・compose.yaml 構成設計 → GitHub Wiki
                       ↓ APIスキーマ確定次第（完全完成を待たず）即座に次へ

【フェーズ3】同時並行
  Technical Writer  → APIドキュメント先行整備（domain-model.md をもとに）
  Backend Architect → compose.yaml 実装・DB設計・API実装・ユニットテスト作成（git worktree）
  Frontend Developer → MSWでAPIをモックして実装・ユニットテスト作成開始（git worktree）

【フェーズ4】随時
  Code Reviewer → PR が来たら即レビュー（Backend・Frontend を並行レビュー）
  PR起票者      → Approve後 `gh pr merge` でマージ

【フェーズ5】統合後
  Frontend Developer → Playwrightでe2eテスト作成・実行（docker compose up 済み環境で）
```

### PR運用ルール

- **Frontend Developer / Backend Architect** は作業完了後、必ずPull Requestを作成する（直接mainへのpushは禁止）
- **PR提出後は即座に次のタスクへ着手する。** レビュー完了を待って手を止めることは禁止。Code Reviewerからの指摘が届き次第、そちらを最優先で対応する。
- **Code Reviewer** は作成されたPRを必ずレビューする
- レビュー未実施のPRはマージしない
- ApproveされたPRのマージは **PR起票者自身** が `gh pr merge <PR番号> --merge` でCLI経由で実行する（GitHub Web UIからのマージは行わない）
- PRのタイトル・説明には変更内容と目的を明記する
